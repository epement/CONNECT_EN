Frequently Asked Questions about the CONNECT program and their answers

Q. My first impression (or rather, a question): Why are there so many
menus? Who is this shell designed for? a hacker who uses nothing but a
memory monitor and an editor to crack EXE files, or a regular user who
needs to unzip, zip, and copy files from one place to another, wants a
menu to pop up right in front of them at the press of F2, and needs to
quickly browse through archives?

A. The integrated CONNECT shell is designed primarily for programmers.
It was originally written as a shell for launching cross-development
tools and communicating with a controller via a COM port. Today, without
losing its original capabilities, it has evolved into a universal
workstation for programmers working in DOS. CONNECT makes it easy to
write program code, run a compiler on it, fix any errors the compiler
finds, manage files and archives using a powerful file manager, and write
program documentation using a robust and user-friendly text editor. You
can even dabble in a bit of "hacking," if you like. As for its
capabilities regarding controllers connected to a serial port ... well,
that's a separate conversation for those who actually need that specific
functionality.


Q. Why is there no cursor in the command line within the file manager?

A. In CONNECT, the file manager is structured somewhat differently than
it is in Norton Commander or similar programs. In CONNECT, the command
line is a separate window; to work with it fully, including using the
cursor, you must first switch focus to that window. To do this within the
file manager, press Ctrl-<Down Arrow> (Ctrl-Dn). In general, the
Ctrl-<Arrow> key combinations allow you to switch focus between the
various panels of the file manager. Furthermore, each of the panels
(including the command line) may have its own "Exec options"; this is
something you should keep in mind both when configuring and when working
with CONNECT.


Q. Why is the cursor not visible at all in any of CONNECT's windows? And
in <Replace> mode? When the cursor is a solid block, is it visible then?

A. This can happen when transferring CONNECT from one machine to another
in its entirety, that is, along with all the `connect.*` files. In such
cases, the cursor parameters configured for a machine with an EGA video
adapter do not match the parameters required for a VGA adapter. To
resolve this, go to `Options` > `Cursor` and adjust the cursor settings
so that it is visible in all modes.


Q. I'm testing how the `BRIEF` and `FULL` display modes work [and]
everything is fine. Then I test the `Descriptions` feature, and it's a
bust. "Okay," I think to myself, "I probably forgot to enable something
again."

A. That is indeed the case. The mode for displaying and editing file
descriptions is enabled within the `FULL` panel mode using the command
<Show file descriptions> (Ctrl-P). Furthermore, to ensure that file
operations (such as copying, renaming, or deleting) also affect their
corresponding file descriptions, you must enable the option `File
Manager` > `Copy/rename/delete descriptions`.


Q. What is the quickest way to get a handle on CONNECT's capabilities?

A. Unfortunately, I cannot recommend reading the documentation for one
simple reason: it doesn't exist. CONNECT is a non-commercial product, and
we simply lack the time and energy to write a manual; even if we *did*
write one, it's unlikely *you* would have the energy to read it! :)

Therefore, the best approach is to explore the main menus and the
window-specific context menus (which can be accessed in any window by
right-clicking or pressing <Alt-F10>) to see what actions are available.
In confusing situations (or better, always!) press F1 to bring up
context-sensitive help.  Everything (or almost everything) is explained
there. The most important thing to remember is that in CONNECT, you can
do almost anything you want; you just have to figure out *how* to do it.


Q. What is the fastest way to configure CONNECT?

A. Generally speaking, if you have obtained the original CN.ZIP archive,
it already contains at least a partial configuration for everything,
leaving you only to fine-tune it to your specific needs. Specifically,
the utilities included in the original archive, prg2txt.exe and
txt2prg.exe, are designed for this purpose: they allow you to convert
configuration files for viewers, File Associations, and the Program List
from binary format into plain text and back again.

Furthermore, it is crucial to correctly configure the temporary directory
right from the start. In the provided configuration file, this setting
(Setup | Swap file directory) is set to `$TEMP`. This means it inherits
the value of the `TEMP` environment variable, which must be defined
[usually in AUTOEXEC.BAT] for this setting to work. Alternatively, you
can simply type the name of the temporary directory directly into this
field.

Q. Why does CONNECT occupy a full 15 kilobytes of memory when launching
programs? Even Norton Commander only occupied 12.

A. Obviously, this is because CONNECT was launched via the batch file
`cn.bat`, which contains the command `cn %1 /b15000/s`. This allocates
15,000 bytes for the buffer of the resident screen "snipper" (capture
utility). If the amount of resident memory occupied is critical, you can
disable the loading of the snipper via the menu option `Exec options |
Load Snipper`. You can also reduce the size of the snipper buffer, or
omit the buffer specification entirely; in that case, a default size of
2,048 bytes will be set. If a free block is available in the UMBs on your
machine, the Snipper buffer will be allocated there.

The [three] radio buttons under `Main` control the behavior of the core

In `Swap` mode, the following components remain in memory: the CN.EXE
stub (which serves to launch CONNECT.EXE), the Snipper (if it is
currently loaded), and approximately 4 KB of the main program's code. The
remainder of the main program, along with all its data, is swapped out to
disk, EMS, or XMS, depending on which type of memory is currently
available. Crucially, the portion of XMS or EMS memory occupied by
CONNECT's data is *not* released; instead, it remains allocated and
continues to be utilized upon returning to CONNECT. Provided there is a
sufficient amount of free EMS or XMS memory available, `Swap` mode
represents one of the fastest and most efficient methods for launching
external programs. However, you cannot load memory-resident programs
(TSRs) while in this mode; doing so would prevent the main program from
reloading correctly upon its return, and the allocated EMS or XMS blocks
would fail to be released. To ensure that edited files are preserved when
launching in `Swap` (or `None`) mode, a particularly important
consideration if the program being launched will be working with those
files, you must enable the `Save editor files` option.

In `Unload` mode, only CN.EXE remains in memory, along with the Snipper
(if the `Load Snipper` option is enabled). In this scenario, CN.EXE is
responsible for launching the selected program. CONNECT.EXE is completely
unloaded from memory, closes all open files, and releases any allocated
EMS or XMS. Upon returning from the launched program, CN.EXE relaunches
CONNECT.EXE, which then restores its state by reading from a `*.dsk`
file.

To ensure that the exact state prior to launching the program is
restored, the `Save desktop` option must be enabled. `Unload` is the
slowest launch mode, especially on systems with large amounts of extended
memory, but it is also the most reliable. In this mode, it is possible to
load memory-resident programs; these will be loaded after CN.EXE, and if
you subsequently exit CN.EXE, a "hole" (unusable gap) will remain in
memory.

Option `None` is the fastest launch mode. In this mode, only available
free memory (the size of which is displayed by the indicator in the
bottom-right corner) and the overlay buffer are released; the size of the
overlay buffer is configured via the menu path `Options > Overlay >
Overlay buffer size`. If the program being launched requires no more
memory than this, this constitutes the most optimal execution method for
it.  Naturally, resident programs cannot be loaded while operating in
this mode.

The `Message` option, if enabled, redirects the standard output of
the launched program to a file, which is subsequently displayed in the
"Message" window.

The `Load COMMAND.COM` option, when enabled, loads the COMMAND.COM
command interpreter (or an alternative interpreter [like 4DOS, NDOS, or
FreeDOS] specified by the COMSPEC environment variable) prior to
launching the program.  This is useful when launching programs located
outside the current directory and is essential for executing .BAT files;
however, in the latter case, this option is enabled automatically.

The `Swap to disk` option serves to force the creation of a swap file on
disk (rather than in extended memory). This can be beneficial when
launching programs that have a greater need for that specific memory
resource.

Q. What is SNIPPER, and how do I use it?

A. SNIPPER is a resident utility for "capturing" text screens, allowing
you to select any area of the screen and place it into a buffer. From
this buffer, the information is automatically transferred to the
clipboard.  Snipper is activated by simultaneously pressing the
`Alt-Ctrl-W` keys.  When this happens, a cursor appears on the screen,
controlled by the arrow keys. After selecting the starting position for
the area to be captured, press `<Enter>`; then, use the highlighted
selection area to mark the entire region of the screen you are interested
in, and finally, confirm your selection by pressing `<Enter>` again.

Pressing the `<Esc>` key allows you to cancel the selection process at
any stage. An audible beep indicates that the Snipper buffer has
overflowed.  he Snipper is accessible both within CONNECT itself and
within any text-mode programs launched from it.


Q. Why is it that when copying files to floppy disks using the `Optimal
fill` mode, the actual amount copied is sometimes significantly less than
what the disk could hold? Recopying the files usually resolves the issue.

A. This can occur if the file copying process includes copying their
associated file `descriptions` [referring to the hidden file named
DESCRIPT.ION used by 4DOS and supported by CONNECT]. In this scenario,
the actual amount of data being written to the floppy disk is greater
than what the "optimal fill" algorithm calculates; consequently, the next
file selected for copying turns out to be too large to fit. You can
resolve this issue either by recopying the files or by disabling the
option found under `File Manager > Copy/rename/delete descriptions`.


Q. I found a design flaw in CONNECT: I opened a window and pressed `F6`,
which worked fine, but I couldn't switch back from the File Manager using
either `F6` or `Shift-F6`. I consider this a major oversight.

A. This is neither a bug nor a design flaw; rather, it is a deliberate
choice intended to preserve the familiar keyboard layout. The `F5` and
`F6` keys, which are already assigned specific functions within the File
Manager, have their window-switching functions duplicated by the `Alt-Z
(Zoom)` and `Alt-N (Next)` key combinations, respectively. Additionally,
you can switch between windows using the `Ctrl-Tab` and `Shift-Ctrl-Tab`
key combinations.


Q. I run my own BBS, and users often bring floppy disks with them to copy
files they are interested in. I've heard that CONNECT offers a very
convenient way to handle this, but I'm not sure how to do it.

A. Indeed, CONNECT features mechanisms that are specifically designed to
make this process very convenient. To utilize this feature, you simply
need to have the list of files that the user wishes to download saved as
a separate file. This can be done by copying a selection directly from
the file list.  The only requirement is that the files in the list must
be named in the format `filename.ext`.  You must place this list onto the
clipboard, for instance, by loading it into the built-in viewer or
editor, selecting the desired block of text, and executing the `Copy`
command `(Ctrl-Ins)`. After this, executing the `Paste` command
`(Shift-Ins)` within the file manager panel will result in any files that
exist in *both* the current panel directory and the clipboard being
highlighted.

At this point, all that remains is to copy them to floppy disks using the
`Optimal fill` mode. If the files on the BBS are organized in a complex
directory tree, you may wish to execute the `Unfold subdirectories`
command `(Alt+B)` on specific branches *before* highlighting. This allows
you to process an entire branch, which this command expands into a
single, flat list all at once.  However, when processing excessively
large branches of the tree, you may encounter memory limitations in
generating this list. After one branch has been processed, you should
proceed to the next.


Q. How can I configure the file manager so I can open archives by
pressing the `<Enter>` or `F3` keys?

A. Generally speaking, the latest distribution version is already
pre-configured to open archives via the `<Enter>` key. Here is how it is
done: The action triggered within the file manager when you press
`<Enter>` on a specific filename is configured in the `File associations`
dialog. You can access this dialog either through the file manager's
local menu or by pressing the hotkey `Ctrl-X`. To designate a file
(defined by a specific mask) as an archive, you must enter "$Arc" into
the "Program path" field.

Configuring the `<F3>` key to view archives is done in an analogous
manner. The `Viewer list` dialog can also be accessed via the file
manager's local menu or by pressing the hotkey `Ctrl-V`.  Additionally,
the `Viewer list` dialog may contain the macros `$View` and `$XView`,
which invoke the built-in viewer in `Text` and Hex modes, respectively.
To attempt to open *any* file as if it were an archive, regardless of its
file type, simply place the panel cursor on the file and press the keys
`Alt-A` [archive].


Q. What are those strange strings written in the fields for `Filename
mask`, like `$copy("\*.+\*","(")`, and `Line num mask: $copy("(",")")`,
used for Turbo Assembler and other programs?

A. These are masks used to extract [substrings] from the output lines
returned by the assembler (or other programs writing to `stdout`) the
name of the file being assembled and the specific line/column number
where an assembly error occurred. If the assembler's output has been
intercepted (which occurs when the `[X] Message` option is enabled), you
can place the cursor on that  line in the Message window and press
`<Enter>` or `<Ctrl-Enter>` to open the editor or viewer, [moving the
cursor] directly to the relevant location in the file.

Detailed instructions on how to construct these masks can be found in the
Help documentation. To test their functionality, the CONNECT distribution
includes a dedicated utility called `filter.exe`.  It is also useful to
examine how these masks are configured for other programs, such as `grep`
[a line-oriented search utility not included with CONNECT].


Q. I just downloaded a new version of CONNECT. Which files should I keep
from my previous installation, and which ones should I take from the new
archive?

A. From the archive, you should take all modified `*.exe` files, the
`*.hlp` file (if it has changed), and carefully read the `whatsnew.txt`
file. That file will specify if any other files require updating.
Replacing almost all of the files is generally a relatively painless
process.  The purpose of each configuration file is explained in the
`files.bbs` file located in the original `CN.ZIP` archive.

Your existing `*.ext`, `*.prg`, and `*.vie` files should be retained, as
these store the settings for your external programs. If, after doing so,
the colors in the new windows appear incorrect, you should either
reconfigure them via `Options | Colors` or copy the `connect.cfg` file
from the original distribution package [overwriting the existing
"connect.cfg" file which is responsible for the bad color scheme; you
may want to keep it under a different filename for comparison].


Q. What are the limitations of the text editor? How is it structured?

A. The text editor is structured as follows: The text being edited is
stored as lines within a swap file, stored either on disk or in expanded
memory.  Pointers to these lines are stored in conventional memory as an
array; this array can hold a maximum of 32,000 elements, provided there
is sufficient memory available to store them. Additionally, arrays for
the `Undo` and `Redo` buffers are stored in memory.

The maximum line length is 256 characters [!]. Longer lines are split
into multiple segments and wrapped accordingly. Thus, depending on the
average line length in the file and the amount of available memory,
the editor is capable of loading files ranging in size from 0.5 to 1 MB.
The hex editor is capable of editing files up to 7FCCFH (523,471) bytes
in size. If, during the process of loading a file into expanded memory,
there is insufficient space available, a swap file is created on disk,
and the expanded memory is cleared of its swap data.


Q. What are the `Run` and `Debug` menu items for? Can I use CONNECT to
debug my own programs?

A. These menu items, along with the `User > Load icon` command, are
designed for use with the compiler for the built-in language. This
language is used to define serial communication protocols [the original
purpose and first module of CONNECT] and to control the text editor
itself. More detailed information regarding this language, its
capabilities, and its specific features can be found in the help
documentation under the topic heading `Connect Language`. CONNECT
provides a comprehensive set of tools specifically for debugging programs
written in this built-in language. It is not possible to debug any other
types of programs using CONNECT's debugging tools.


Q. Is it possible to release a "Lite" version of CONNECT that omits the
built-in language and debugger? After all, they probably consume a
significant amount of memory and disk space, and and most users don't
need them.

A. Of course, nothing is impossible for us in this regard, but we do not
intend to do so. First, this would entail maintaining two versions of
such a large program simultaneously, and we are already barely managing
with just one.  Second, these tools are deeply intertwined with the other
parts of CONNECT (specifically the editor), and removing them would
inevitably trigger a whole cascade of errors. Third, CONNECT is an
overlay-based program; the majority of the code related to the language
and debugger resides in the overlay file and doesn't get in anyone's way.
Furthermore, while very few people actually need to program protocols,
editing text *within* the program is not nearly as rare a use case. And
this can be done quite effectively using the built-in language tools.

In the original CN.ZIP archive, in the CBS directory, there are several
sample programs: some [are used] for defining protocols and others for
text editing. These programs are thoroughly commented, so any programmer
should have no trouble understanding them.  Ultimately, many editors and
terminal emulators incorporate elements of a programming language, but
how many of them also offer such convenient debugging tools?


Q. I regularly find leftover files in CONNECT's temporary directory.
Where do they come from and what should I do with them?

A. These are temporary files generated by the editor, the archive viewer
executable, and other components of CONNECT. Under normal shutdown
conditions, these files should be deleted automatically. However, if the
operating system crashes, if the computer is powered off abruptly, or if
a program launched from within CONNECT terminates abnormally, these files
may remain open or left behind, which may lead to file system errors.
Therefore, I recommend adding the following lines to your `autoexec.bat`
file:

  :: [Set a variable matching the TEMP directory used by CONNECT.]
  SET TEMP=C:\TEMP
  ::
  :: [Set Archive attribute, clear Read-only, System, Hidden attributes]
  ATTRIB -R +A -S -H C:\TEMP\*.*
  ::
  :: [Delete all files in the TEMP directory used by CONNECT.]
  ECHO Y | DEL C:\TEMP\*.*
  ::
  :: [Automatically run CheckDisk on drive C at each reboot.]
  ECHO N | CHKDSK /F C:

This will clear your temporary directory every time the machine is
rebooted, helping you avoid many potential issues, and not just
[problems] related to CONNECT.


Q. Why doesn't CONNECT support extended video modes?

A. Extended video modes cause more trouble than they are worth. First,
they are non-standard: specific settings for each mode must be
individually configured for every video card, which is inconvenient.
Second, standard mouse drivers rarely function correctly in these video
modes, and using non-standard mouse control methods can lead to
additional complications. And finally, I don't know about you, but I
personally find it difficult to read text in 80x50 mode, let alone in
132x80 mode!

The version of CONNECT dated November 4, 1997 [and later] *does* support
extended video modes. They can be enabled by executing the command-line
macro `$VM(nn)`. Simply type `$VM(nn)` at the command prompt, where `nn`
represents the decimal value of a video mode supported by your specific
video system, and CONNECT will switch to that mode.


Q. When will CONNECT feature a "graphical" mouse cursor?

A. I don't think it ever will. That, too, would create more problems than
it solves. The most significant drawback is that it requires
reprogramming the character generator. Consequently, windows such as the
`ASCII Chart` would end up looking distorted or garbled. Ultimately, it
offers no real practical benefit.


Q. Why is the `clipboard` necessary, and how do you use it?

A. The clipboard is a tool for exchanging information between various
windows and input fields within CONNECT. [It is NOT related to the
Windows Clipboard.] Almost all windows and input fields support `Copy`
and `Paste` operations. Moreover, windows [can] place information on and
retrieve it from the clipboard in a format that is usable by other
windows, as well as by themselves. For instance, the calculator will not
accept information from the clipboard if that information is not a
number. The clipboard window itself is always quickly accessible for
editing (via `Alt-Ins`) and, from this perspective, functions just like a
standard editor window. Anything highlighted within it can be placed into
another window or input field.  Fundamentally, the clipboard is precisely
what makes CONNECT an integrated environment rather than merely a
collection of separate programs. To determine which windows allow you to
place information into the clipboard (and which ones do not), you simply
need to try doing so.  In all likelihood, you won't be disappointed.

If you find yourself out of luck, where CONNECT will not let you `copy`
[data] from a particular window, you can always make use of the "Snipper"
tool [which should work when the Clipboard will not].


Q. Why are macros necessary, and how do you use them?

A. A macro is a recorded sequence of menu commands or keystrokes that can
be played back once or multiple times; it can be invoked from a list or
via an assigned hotkey, as well as edited or deleted. A macro can even
call itself recursively any number of times. The macro playback process
can be interrupted by pressing the `<Esc>` key. Macros can be created
manually using the macro editor (`User > Macro list > New`) or by
directly recording a sequence of keystrokes (e.g., `Record macro
(Alt+Equal), Stop recording (Alt+Minus)`).

To play back a macro, open the `User > Macro list` dialog, select the
macro you just recorded (it will be the last one in the list), and
execute it using the `Run` command (to run once) or `Repeat` (to run a
specified number of times).  If you plan to use a macro frequently, it is
best to give it a descriptive name and assign a hotkey so that it can be
invoked quickly and conveniently later on.


Q. I downloaded the new version of CONNECT, but it froze immediately.

A. This may be due to a change in the `*.dsk` file format. You simply
need to delete that file. [WARNING: This does not always work. I have
found that CONNECT will not load or may hang if CONNECT.DSK does not

Q. How can I make the archiver's progress visible while it is running?

Q. CONNECT is calculating directory sizes incorrectly.

A. In CONNECT, directory sizes are displayed in two locations: in the
file size column on the file panel (provided the "Display directory
sizes" option is enabled), and in the information window that appears
when you press F3 while a directory is selected. Note that in the
information window, the *total size* of the files is displayed in
parentheses following the file count, while the *actual disk space
occupied* by those files is shown in the "Total size" line. Sizes may be
displayed in bytes or kilobytes (1024 bytes). The directory sizes
displayed on the file panel are retrieved from the `treeinfo.cn` file and
may not always reflect the current reality. This data is updated whenever
the directory tree is rescanned or when directory sizes are explicitly
calculated using the F3 key. Dynamically tracking changes in directory
sizes is too complex and unreliable; therefore, we opted for the current
approach. The directory sizes displayed in the panel represent the total
size of all files within that directory and its subdirectories.


Q. How do I configure external viewers?

A. This is a complex topic. When configuring external viewers as well as
`File Associations` (the actions triggered by pressing `Enter` on a
filename in the file manager, or `Shift-F9` in the editor) and the
`Program List` (accessed via `F9`), [CONNECT's] unique macro language is
employed to control the execution sequence and results of external
programs.

In the "Program path" field within the external program configuration
dialogs, you can enter the names of [one or more] programs, separated by
a semicolon (;). [These commands will execute in that order. Note that
these are commands to run, not a strict directory path like "c:\DOS".]

Similarly, in the "Command line" field, the parameters for the launched
programs can be listed, also separated by semicolons.  Parameters
separated by semicolons are interpreted according to their sequential
order: the first parameter (everything up to the first semicolon)
corresponds to the first program, the second parameter corresponds to the
second program, and so on. If a program parameter itself needs to contain
a semicolon, you must replace it with a double semicolon (;;). [See the
example for "Add Path" in the Program List.]

Parameters may contain the following macros:

$PATH, $NAME, $EXT, $LEFT, $RIGHT, $OPPOSITE, $CN, $SWAPDIR, $DIRARC,
$REPORT, $WORD, $PROMPT, $SEL, $MSG, $NOMSGWIN, $USERSCR, $KBD, $MEM,
$NOREDRAW, $NOREREAD. 

Program names:

$VIEW, $XVIEW, $DBASE, $EDIT, $XEDIT, $ARC, $PANEL, $ICON, $HELP. 

The meaning of these and several other macros is described in detail in
the Help documentation and in the WHATSNEW.TXT file.

In addition to these fields, there are also two specific fields:
"Filename mask" and "Line num mask." These parameters tell `CONNECT` how
to locate specific lines within the file currently being processed by the
launched program---specifically, those lines for which the program has
output information [sent] to standard output, which will be redirected to
the "Message" window. This information might consist of compiler
diagnostics regarding errors found in the program's source code, output
from a utility searching for specific text strings within program files,
and so forth. An example demonstrating the use of this mechanism is the
execution of Turbo Assembler:

Q: I open a file from an archive using the viewer. Then I switch to the file manager and work there. When I return to the viewer and attempt to view the text, a disk error occurs.

A: When you view a file from an archive, it is extracted to a temporary directory and then loaded into the viewer. When you exit the archive, all files that were extracted from it for viewing are deleted; however, the viewer is unaware of this and continues to display the text from its internal buffer. The error occurs when you attempt to view text located beyond the limits of this buffer, as the program tries to read from a file that has already been deleted. The viewer will behave in the exact same way if you simply delete the file currently being viewed. While it would be technically possible to send a notification to every open desktop window whenever a file is deleted, doing so would significantly slow down the deletion process?especially during batch deletions.

Q. Why doesn't the text viewer display the current line number?

A. Because the viewer does not read the file line by line; instead, it opens the file at a specific byte offset?the same offset that was saved in the history log after the file was last opened. Similarly, when you press Ctrl-PgDown, the viewer does not read the file line by line; rather, it jumps directly to the end of the file (leaving a small buffer to ensure the final screen of text is fully visible). Consequently, maintaining a line counter would require a completely different?and significantly slower?operational algorithm for the viewer. We consider the performance trade-off required to implement such a counter to be unjustified.
